Decorations

These are the element that truely emphasize the content. Ther are several decoration type. It can be definitions for module frameworks or text behavior (size, color). These also define the working directory, where lies the graphics. Decorations are not depending on anything. These can exist in the database without any link to the reste of it. These are stored in different tables depending on the type. Tese are not bound to a website in particular. So a theme can use it even if it was created for another theme of another website on the same server. Please note that it is encouraged to produce CSS stylesheet (using website administration) to minimize the downloads. A MWM CSS stylsheet can easely have more than 1000 lines.

Différent decoration type

Decoration have been named to ease comprehenssion.

1_div

It is a DIV (HTML) which attributes are customizable. It is often used for tooltips.

Elegance

This one use a set of DIVs that surround the content. It is organized in line and column to ease comprehenssion.

Exquise

This one use a set of DIVs that surround the content. It is organized in line and column to ease comprehenssion. The difference is this decoration use more DIVs.

Elysion

This one use a set of DIVs that surround the content. To this it adds a set of DIVs that will be placed over the content DIV. It is organized in line and column and INternal and EXternal types to ease comprehenssion.

Les thèmes (ou skin)

These elements are like some sort of directories that link to decorations. These also define the use of CSS stylesheet files, logos (blason module), or some element of the administration panel. Each theme is linked with only one website. There are no sharing option for themes. In the mean time use of scripts facilitate the copy to another website. To understand this system you must understand how the website works. When a page is browsed, the engine will search for module to render. Each module defnine if there is a decoration to render and what is its number. The theme contains definitions that are compatible with this system.

Ex:

> block_01_name 'one_block_001' block_01_text 'one_texte_001'
block_02_name 'one_block_002' block_02_text 'one_texte_002'
etc.
